## Context

AuctorForge already has a branded startup screen with create, import, recent-project, management, and model-settings entry points. It also has local project lifecycle APIs, file APIs, workflow state, and model settings. What is missing is a first-run narrative that makes the safe path obvious to writers who are deciding whether to trust the app with unpublished work.

The onboarding design must fit the existing quiet launcher rather than introduce a marketing page. It should guide action inside the product surface: try fictional content, create/open a project, understand local storage, and understand when model-provider calls may happen.

## Goals / Non-Goals

**Goals:**

- Make the startup screen explain the first safe trial path in one glance.
- Provide a sample-project entry point that does not require private manuscript text.
- Show local-storage and model-request boundaries before a user commits real material.
- Keep existing create/import/recent-project flows intact.
- Make empty and misconfigured states understandable to writers.

**Non-Goals:**

- Replace the launcher with a marketing landing page.
- Change existing project file formats or workflow folder names.
- Require remote model configuration before a user can inspect the app.
- Build full export/backup controls in this change.
- Build the long-term template library in this change.

## Decisions

### 1. Add onboarding as a compact startup module

The first-run guidance should live on the existing startup surface as a compact module near the primary actions. It should not be a blocking modal. Writers should still be able to create, import, manage projects, or configure models immediately.

Alternative considered: a multi-step wizard before the launcher. Rejected because it delays returning users and makes the app feel heavier before trust has been earned.

### 2. Treat sample content as a project creation path

The sample-project action should create or duplicate a fictional project through the same project lifecycle boundaries used by normal projects. That keeps file ownership, registry behavior, and workbench routing consistent.

Alternative considered: a static preview-only demo. Rejected because it would not exercise the real file tree, editor, chat context, or workflow surfaces that users need to evaluate.

### 3. Keep model-request transparency close to model actions

The startup surface should state that local editing can be explored before enabling a provider. Model settings and any future model-call preview should explain provider, endpoint, and request scope where the action happens.

Alternative considered: putting all privacy detail in docs only. Rejected because writers need the boundary at the moment they choose whether to continue with real material.

### 4. Make onboarding state local and dismissible

The onboarding module may be dismissible through client-side UI state. Dismissing it must not hide core safety information permanently from docs or model settings.

Alternative considered: storing onboarding completion in project files. Rejected because onboarding is an app-level preference, not a manuscript artifact.

## Risks / Trade-offs

- [Risk] Too much safety copy can make the launcher feel anxious. → Mitigation: keep the module concise and link to privacy docs for detail.
- [Risk] Sample project generation could diverge from normal project initialization. → Mitigation: route through the existing project lifecycle and add tests around registry, files, and workbench routing.
- [Risk] Users may assume no remote calls ever happen. → Mitigation: phrase local-first claims carefully and tie remote-call warnings to model-provider configuration.
- [Risk] Dismissed onboarding may hide useful guidance from a later user on the same machine. → Mitigation: keep safety copy available in settings/docs and allow a reset or help entry later.

## Migration Plan

No data migration is required. Existing projects and project registry entries continue to work. If local onboarding-dismissal state is introduced, it should default to showing onboarding for users without the key.

Rollback is removing the onboarding module and sample-project entry point; existing projects remain compatible.

## Open Questions

- Should the first sample project be bundled as static files, generated by server code, or copied from `skill-packs` plus small fictional content?
- Should sample projects be created in a user-selected folder or a clearly named default local folder?
- Should dismissal be stored in browser localStorage or server-side user configuration?
