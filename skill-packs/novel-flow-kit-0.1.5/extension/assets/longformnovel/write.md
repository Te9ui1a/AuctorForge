# Workflow: Write 


## 步骤 0: 一次性初始化 (One-time Setup)

*   **目标**：以主角的初始状态作为整个创作阶段的基准。（拆件初始化时已创建相关文件，本步仅做更新与链接。）
*   **动作**：首次创作前，根据 `XX卷_章纲.md` 的开局设定：
    1.  **更新初始状态**：在 `.novelkit/memory/character_state.md` 中记录/更新主角第 0 章（正文开始前）的初始状态。
    2.  **链接到 PROJECT**：确认 `PROJECT.md §7.2 Character Snapshot` 已写入指针 `-> .novelkit/memory/character_state.md`；若未写入则补上。


---


### 步骤 1：写作前置检查 (Pre-Writing Check)
*   **目标**：确保动笔前的所有信息输入是完整且准确的。
*   **动作**：在撰写 **每一章** 正文前，必须检查并整合以下关键信息：
    1.  **风格约束**：
        *   `1-边界/1.2_文风.md` (基础文风 — 路径已注册于 `PROJECT.md §5`)
        *   `1-边界/1.5_微观节奏拆解.md` (如有：导演手册，场景调度/镜头语言)
    2.  **红线底线**：`docs/MASTER.md` (避坑指南 — 路径已注册于 `PROJECT.md §6`)
    3.  **设定支持**：
        *   查阅 `PROJECT.md §2`（世界观索引）和 `§3`（角色索引），按指针读取对应的世界观和角色档案文件
        *   `2-设定/2.3_金手指设定.md` (如有：道具说明书，若本章涉及使用)
    4.  **当前状态**：`.novelkit/memory/character_state.md`（路径注册于 `PROJECT.md §7.2`，确保获取最新状态）
    5.  **伏笔与进度**：
        *   `.novelkit/memory/foreshadowing.md`（路径注册于 `PROJECT.md §7.4`）
        *   `PROJECT.md §8`（当前阶段与待办）
    6.  **本章指令**：`3-大纲/XX卷_章纲.md` 中对应的本章章纲部分。
    7.  **上下文参考**：读取 **上一章正文** (确保剧情连贯、语气承接)。
    
*   **目的**：防止逻辑BUG、风格漂移和设定遗忘。

### 步骤 2：正文撰写与AI辅助 (Writing with AI)
*   **输入**：步骤1中整合的全部信息。
*   **动作**：
    *   **分场景生成**：严格按照章纲拆解的场景顺序，**逐个场景**进行撰写。
    *   **字数锚定**：建议单章总字数在 **2300-4000 字** 之间（用户可根据平台要求自行指定范围）。
    *   **文风执行**：严格执行 `1.2_文风.md` 的要求，注重“展示而非讲述”(Show, Don't Tell)，强化感官描写。
    *   **结尾钩子**：确保章节末尾包含章纲中设计的悬念或钩子。
    *   **创作范围**：严格遵守章纲剧情范围，不得超出。
*   **输出产物**：`4-正文/第X章_草稿.md`。

### 步骤 3：完稿后校准 (Post-Writing Calibration)
*   **动作**：
    1.  **严格审查（五项检查）**：
        *   **字数检查**：确保单章字数在2300-4000字之间（使用脚本 `scripts/count_chinese_words.py` 进行统计），可以超出 4000 字，但是不能低于 2300 字。
        *   **大纲符合性检查**：对照 `XX卷_章纲.md`，检查本章核心事件、场景和结尾钩子是否完全按计划执行。
        *   **文风检查**：对照 `1.2_文风.md` 和 `docs/MASTER.md`，检查是否存在AI写作痕迹。
        *   **接续检查**：对照上一章，检查剧情、时间、地点是否无缝接续。
        *   **金手指合规性**：(若涉及) 检查表现形式是否符合 `2.3 金手指设定.md`。

    2.  **显式输出【完稿自检卡】**：注意：如果五项检查有一项不通过，必须重新创作，或者进行相应修订
    3.  **AI味局部修补**：若审查报告或服务端补充给出“局部改写任务”，只改命中的句子、段落或场景片段；非目标段落必须保持原样。禁止用同义套话替换禁用词，必须改成动作、具体后果、感官细节或对话潜台词等写法策略。
    4.  **审查后定稿**：当用户要求“按审查报告修改 / 局部修补 / 生成定稿”时，先读取本章草稿和本章审查报告。只修改审查报告点名的问题句、段落或场景；非目标文本保持不变。输出 `4-正文/第NNN章_定稿.md`，不要跳到下一章。
*   **目的**：保持设定与正文的实时同步，确保单章质量达标。

### 步骤 4：伏笔、状态与进度更新 (Foreshadowing, State & Progress Update)
*⚠️ **CRITICAL STEP**: 此步骤决定了记忆的连续性。绝对不可跳过！*

*   **状态回写**：根据本章发生的实际剧情，**立即更新**以下文件：
    *   `.novelkit/memory/character_state.md` → 角色状态变化（升级/道具/关系）
    *   `.novelkit/memory/foreshadowing.md` → 新增或收回的伏笔条目
    *   `PROJECT.md §8` → 将本章标记为已完成，填入下章任务
*   **执行确认**：必须明确输出"已更新 memory 文件"的反馈，方可结束本章创作循环。

---

## 🚫 循环阻断 (Loop Break)
*   **动作**：在完成上述所有步骤（1-4）后，**必须暂停**。
*   **状态检查与分支选择**：
    1.  **正常推进**：
        - 检查当前卷 `XX卷_章纲.md`。
        - **有剩余章节**：询问 “第 X 章完成。是否继续第 X+1 章？”
        - **本卷已完**：提示 “本卷完。请调用 `novel-outline` 规划下一卷。”
    2.  **剧情偏离/修订**：
        - 若本章剧情与章纲差异较大，或用户表示“有了新想法”。
        - **引导**：“检测到/你提到剧情有变化，是否调用 `novel-outline` 的【动态修订】模块，重做后续大纲？”
*   **等待**：等待用户明确指令，严禁自动开始下一章。

2.  **质检通过后** → 继续下一章的 `novel-write` 循环。
