# 长篇小说工作台 UI 设计基准

> 状态：当前生效
> 用途：后续所有前端 UI 设计、页面改版、组件风格调整，默认都先参考本文件
> 适用范围：首页、项目中心、工作台外壳、文件树、聊天、编辑器、流程栏、设置弹层、图标、配色、文案语气

---

## 1. 这份文档解决什么问题

本项目已经不是一个临时 demo，也不是单一页面的管理后台，而是一个 **面向长篇小说创作的工作台产品**。

过去的 UI 讨论里已经出现过这些问题：

- 首页太像工具入口，不够像产品首页
- 整体从“乱”进步到了“整齐”，但还不够“好看”
- 页面层次、颜色、图标、文案语气经常在“工具化”和“产品化”之间摇摆
- 工作台、首页、弹层如果没有统一设计原则，后续越改越容易散

所以这份文档的目的不是提供一版短期视觉方案，而是提供一个 **长期稳定的设计判断标准**：

> 以后所有 UI 设计都要回答同一个问题：
> **它有没有让产品更像“一个认真打磨的长篇小说创作产品”，而不是一个临时拼起来的工具后台？**

---

## 2. 产品定位

### 2.1 产品一句话定义

这是一个 **帮助用户把灵感持续推进成完整长篇小说** 的创作工作台。

它不是：

- 普通文档编辑器
- 通用聊天工具
- 单纯的项目管理后台
- 只做设定卡片的灵感笔记本

它是：

- 有首页门面感的创作产品
- 有明确工作流的写作工作台
- 能从设定、大纲、正文、审查一路往前推进的系统

### 2.2 用户第一印象

用户打开产品时，第一反应应该是：

1. **这是一个正式产品**
2. **它是为长篇小说创作专门设计的**
3. **它既能开始新故事，也能继续已有项目**
4. **它有工作台感，但不是冷冰冰的后台系统**

---

## 3. 核心设计原则

## 3.1 产品首页优先，工具入口其次

首页不是项目数据库首页，不是控制台入口。

首页第一职责是：

- 传达产品价值
- 建立气质
- 给出开始入口

首页第二职责才是：

- 承接最近项目
- 让 returning user 高效继续

### 设计判断

如果某个首页改动让“项目管理信息”比 Hero 更抢眼，那么它大概率是错的。

---

## 3.2 外冷内暖

整个产品的视觉应遵循：

- **外层（shell）冷静、专业、克制**
- **内层（editor/content）温和、可久读、像创作空间**

也就是说：

- 顶栏、侧栏、流程栏、设置面板 → 偏冷、偏工具
- 编辑器、只读提案、长文内容 → 偏暖、偏内容

### 设计判断

如果所有区域看起来都一样冷，一样像工具面板，那就不对。

---

## 3.3 主角必须明确

每个页面必须有主角。

- 首页主角：Hero + 产品证明卡
- 工作台主角：当前编辑内容 / 当前创作焦点
- 弹层主角：操作本身，不是背景

### 设计判断

如果每一块都差不多重，页面就只会“整齐”，不会“好看”。

---

## 3.4 写作工作台，不是 issue tracker

虽然工作台会借鉴 Linear 的壳层秩序，但不能做成项目管理软件味道。

避免这些感觉：

- 太像任务管理系统
- 太像开发控制台
- 太像运维面板
- 太像传统 admin dashboard

### 设计判断

工具秩序可以借，但情绪不能借成项目管理软件。

---

## 3.5 统一比花哨更重要，记忆点比统一更高级

优先级顺序：

1. 先统一
2. 再分层
3. 最后建立记忆点

不能为了做“惊艳”而破坏一致性，但也不能永远停在“只是不难看”。

---

## 5. 页面级设计原则

## 5.1 首页（Startup）

### 首页是什么

首页是：

- 产品门面
- 创作入口
- 最近项目承接页

首页不是：

- 项目管理后台
- 项目数据库首页
- 启动器控制台

### 首页信息顺序

正确顺序：

1. Top navigation
2. Hero
3. Product preview card
4. Continue recent projects
5. Start paths / modes
6. Benefits
7. Management entry

错误顺序：

1. 项目中心
2. 新建/导入/管理
3. 最近项目
4. 模式卡

### 首页视觉要求

- 顶栏不 sticky，不漂浮，不像系统工具条
- Hero 是第一主角
- 右侧产品预览卡是第二主角
- 最近项目区必须比 Hero 更克制
- 项目管理入口必须后置，不与主 CTA 同级

---

## 5.2 工作台（Workbench）

工作台是高强度使用区，不需要像首页那样“卖产品”，但必须：

- 稳定
- 清晰
- 可久用
- 有主次

### 设计要求

- 顶栏是系统信息，不是 Hero
- 文件树和流程栏承担扫描效率
- 中央编辑区必须是内容主角
- 聊天要像创作助手，不像普通 IM

---

## 5.3 设置与弹层

### 原则

- 弹层必须是实体，不发虚、不发飘
- 背景遮罩可以半透明，但面板本体必须是实底
- 默认常见笔电高度下不应内部滚动
- 如果需要 fallback 滚动，应该只在短视口下触发

### 特别要求

- settings 面板优先是“可一眼看全”的 dialog
- 不是一个塞在弹层里的滚动页面

---

## 6. 配色体系

## 6.1 总体结构

配色分为 4 层：

- `entry`：首页 / 首页门面
- `shell`：工作台壳层
- `document`：内容/编辑层
- `overlay`：弹层 / 对话框

### 当前核心原则

- 首页可以更有产品感、更有气氛
- 工作台必须克制
- 内容层必须更暖、更安静
- overlay 必须更实、更清晰

---

## 6.2 首页配色

首页应采用：

- 深墨蓝黑为底
- 更清楚的主次亮度
- 主 CTA 用更明确的产品蓝
- 产品证明卡可引入暖纸面感

### 禁止

- 首页继续像单一深色后台
- 首页 Hero 和最近项目同样重
- 所有东西都用一样的低透明度蓝灰叠层

---

## 6.3 工作台配色

工作台应：

- 比首页更克制
- 不追求过强冲击
- 但要有明确壳层 / 内容层差异

### 禁止

- 所有面板都同一视觉重量
- 文件树 / 流程栏 / 编辑区没有主次差异

---

## 6.4 设置面板配色

设置面板本体必须：

- 使用实体底色
- 有清晰边框
- 有明显阴影
- opacity 保持 1

### 不能接受的状态

- 看起来像透明玻璃
- 背后内容穿透感过强
- 面板和 backdrop 混在一起

---

## 7. 字体与排版

## 7.1 字体角色

### 首页

- 标题可更有产品感/编辑感
- 副标题要更像产品文案，不像后台说明

### 工作台 UI

- 继续使用清晰、现代、稳定的 sans 体系
- 不要为了“设计感”破坏工具可读性

### 编辑器

- 内容区域可以更偏阅读型字体
- 但要优先保证长时间阅读舒适性

---

## 7.2 标题规则

Hero 标题必须：

- 可以做大，但不能靠挤出来
- 最多 2–3 行
- 行高不能过紧
- 宽度不能太窄
- 右侧预览卡不能把它挤坏

### 错误信号

- 字重叠
- 为了显得有气势，标题被压得太紧
- 标题宽度只够 1–2 个词

---

## 7.3 正文与说明文字

- 首页副标题应该有呼吸感
- benefits / 说明区块的字数要短
- 工作台说明类文字不应太密、太像后台标签

---

## 8. 图标系统

## 8.1 基本原则

- 所有正式 UI 都用统一图标系统
- 不再混用 emoji、字符图标、临时符号
- 图标统一为线性或清晰 SVG 体系

### 默认建议

- 未来可统一到 **Lucide Icons**

---

## 8.2 图标层级

### 一级：产品/主动作图标

- 首页 CTA
- 主要入口
- 关键模式卡

### 二级：系统/工具图标

- 设置
- 管理
- 文件树操作

### 三级：状态图标

- 提示
- 状态
- badge / warning / success

### 设计判断

如果一个图标看起来像开发期临时拼的，就不该留在正式 UI 中。

---

## 9. 组件级约束

## 9.1 首页 CTA

- 一级按钮只有一个主按钮
- 二级按钮可以存在，但不能抢 Hero
- CTA 必须配统一图标体系

## 9.2 最近项目卡

- 卡主体：选择项目
- 明确按钮：继续进入当前项目创作
- 不能让“看起来像按钮”的文案其实没有动作

## 9.3 Tabs / Pills / Badges

- 工作台里允许高密度，但必须有统一节奏
- badge 只能作为辅助，不应抢主视觉

## 9.4 Dialog

- 用实体面板
- 头部、正文、底部动作区要清楚
- 默认不要内部滚动

---

## 10. 交互与路由设计原则

## 10.1 URL 应该表达页面状态

页面状态不能长期只靠 React 内存状态，否则刷新和回退一定会乱。

### 当前已确定的原则

- `/` → 首页
- `/?panel=manage` → 首页管理区
- `/?projectId=...` → 首页选中项目态
- `/projects/:id/create` → 创作工作台
- `/projects/:id/analyze` → 参考模式工作台

### 设计判断

如果一个页面状态刷新后消失，而用户认为它“应该是一个页面”，那它就应该进入 URL。

---

## 10.2 脏稿保护优先于路由跳转

无论点击、切模式、切项目、回退，都不能静默丢掉 dirty draft。

### 设计判断

- 路由是事实来源
- 但草稿保护比路由切换优先级更高

---

## 11. 动效原则

- 动效只服务于层次和状态切换
- 不用为了“炫”而增加噪音
- 首页可以有轻微门面感动效
- 工作台尽量克制

### 允许

- CTA hover
- 卡片轻微 elevation
- modal/drawer 柔和进入
- Hero 区轻微呼吸感

### 不允许

- 全局大面积漂浮动效
- 强炫技粒子、光效、无意义动效
- 影响写作专注度的持续动画

---

## 12. 技术选型约束（面向未来）

这份设计文档本身是**技术栈无关**的，但为了后续迁移，需要明确适配原则。

### 当前技术栈

- React + Vite + TypeScript + React Router
- 全局 CSS
- 自定义 SVG 图标

### 如果未来迁移

若迁到：

- Next.js
- Tailwind CSS
- shadcn/ui
- Lucide Icons

这份设计文档仍然有效。

### 迁移时的原则

- 不要因为换栈就重置设计原则
- `shadcn/ui` 只做 primitives，不决定产品骨架
- `Lucide` 可以作为默认图标系统
- 首页气质、工作台壳层、内容区、弹层层次必须延续本文件定义

---

## 13. 后续 UI 改动前必须回答的检查清单

任何新 UI 设计开始前，都先回答下面这些问题：

1. 这是首页、工作台、内容区，还是弹层？
2. 这个区域的主角是谁？
3. 它应该更偏产品感、工具感，还是内容感？
4. 它借的是哪一个参考（Claude / Linear / Notion / Mintlify / Vercel）？
5. 有没有破坏“首页产品优先、工作台克制、内容层更暖”的原则？
6. 有没有让页面更乱、更像后台，而不是更像产品？
7. 有没有引入新的图标语言分裂？
8. 有没有只做到整齐，而没有建立主次？

如果回答不清楚，就不要直接改 UI。

---

## 14. 明确禁止事项

### 禁止 1：首页重新退化成工具入口

不能让首页再回到：

- 项目管理抢 Hero
- 最近项目比产品价值更显眼
- 首页第一眼像控制台

### 禁止 2：所有区域都一样重

不能把：

- Hero
- 项目卡
- 工作台侧栏
- 编辑器
- 设置弹层

都做成一样重。

### 禁止 3：再出现半透明发虚弹层

弹层本体必须实体。

### 禁止 4：字符/emoji 图标回潮

正式 UI 不再使用临时符号做主图标。

### 禁止 5：为了“高级”牺牲工具可读性

工作台必须是可用第一，审美第二；
首页可以更像门面，但工作台不能被门面化思路污染。

---

## 15. 一句话总纲

> **首页像产品，工作台像工具，编辑区像创作空间，弹层像实体面板。**

以后所有 UI 设计，如果偏离了这句话，就优先回到这份文档重新校准。
